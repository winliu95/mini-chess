# MiniChess AI Engine - Project 2 Report

## 1. 基礎實作
*   在 `state.cpp` 實作了騎士的移動邏輯。
*   在 `minimax.cpp` 完成基礎的 Negamax 與 Alpha-Beta 剪枝搜尋，並實作了黑白棋視角轉換。

## 2. 系統優化
*   **置換表 (TT) 快取**：利用 Hash 值紀錄算過的盤面，避免重複計算。
*   **時間管理與超時防護**：在搜尋迴圈中用 `std::chrono` 定期檢查時間。一旦快超時，就會將 `ctx.stop` 設為 true 並立刻 `break`，確保回傳的是完整的最佳解，防止引擎超時或下出錯棋。

## 3. 搜尋優化
為了能在 2 秒內算得更深，實作了以下剪枝與排序策略：
*   **PVS**：對第一個走法做完整搜尋，後面的走法只用零視窗快速驗證，沒被打破就直接剪枝。
*   **Null Move Pruning**：當己方優勢夠大時，故意讓對手一步，如果對方還是無法突破 beta，就直接剪枝。
*   **LMR**：對排序在後面的 Quiet Move 減少搜尋深度，如果發現分數不好就直接丟棄，節省時間。
*   **Killer Move**：紀錄之前引發 Beta 剪枝的好棋，在後續其他分支搜尋時優先嘗試，增加提早剪枝的機率。
*   **History Heuristic**：將之前勝出或打平的走法，給予較高的分數，讓 AI 優先走那些「以前走起來比較贏」的棋。

## 4. 估值調整
*   **動態價值重設計**：如果直接套用裁判的規則（城堡=6、騎士=7、主教=8），AI 會在「中局」亂拿城堡去換主教。因此我設定中局使用一般棋力比例（城堡=100），直到殘局（步數 >= 100）才切換回裁判分值來精準判斷和棋。
*   **通路兵加分**：依據兵往前推進的格數（越接近升變底線），給予累加的額外加分，鼓勵 AI 積極升變為后。

## 5. 結果與結論
*   透過上述優化，我們解決了超時與快取被清空的問題，引擎搜尋深度從 13 層躍升至 20 層以上。
